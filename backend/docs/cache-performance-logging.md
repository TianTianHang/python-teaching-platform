# Cache Performance Logging Guide

This guide explains how to view and interpret cache performance logs for the Python Teaching Platform.

## Overview

The cache performance logging system provides real-time visibility into cache operations, helping you monitor cache effectiveness and identify performance issues.

## Log Files

Cache logs are stored in: `backend/logs/cache.log`

For example log entries, see: `backend/logs/cache.log.example`

## Log Types

### 1. Individual Cache Operation Logs (Phase 1)

These logs are generated for **each cache operation** and include:

- **event**: Type of operation (`cache_hit`, `cache_miss`, `cache_null_value`)
- **endpoint**: The view or endpoint name (e.g., `CourseViewSet`)
- **status**: Operation status (`hit`, `miss`, `null_value`)
- **duration_ms**: Operation duration in milliseconds
- **hit_rate**: Current hit rate for this endpoint

**Example:**
```json
{
  "event": "cache_hit",
  "endpoint": "CourseViewSet",
  "status": "hit",
  "duration_ms": 2.5,
  "hit_rate": 0.85
}
```

### 2. Performance Summary Logs (Phase 2)

Generated **every 60 seconds** by Celery Beat, these logs aggregate statistics across all endpoints:

- **global**: Overall statistics (total_requests, hit_rate, avg_duration_ms)
- **endpoints**: Per-endpoint statistics
- **top_slow_endpoints**: Top 5 endpoints with highest average duration
- **top_low_hit_rate_endpoints**: Top 5 endpoints with lowest hit rates
- **alerts**: List of active performance alerts

**Example:**
```json
{
  "event": "cache_performance_summary",
  "period": "60s",
  "global": {
    "total_requests": 10000,
    "hit_rate": 0.876,
    "avg_duration_ms": 8.5
  },
  "alerts": []
}
```

### 3. Performance Alert Logs (Phase 2)

Generated when performance thresholds are exceeded:

- **type**: Alert type (`low_hit_rate`, `high_penetration_rate`, `high_slow_operation_rate`)
- **endpoint**: The affected endpoint
- **value**: Current value
- **threshold**: Threshold that was exceeded
- **severity**: Alert severity (`WARNING`, `ERROR`)

**Example:**
```json
{
  "event": "cache_performance_alert",
  "type": "low_hit_rate",
  "endpoint": "ProblemViewSet",
  "value": 0.65,
  "threshold": 0.8,
  "severity": "WARNING"
}
```

### 4. Request-Level Cache Performance Logs (Phase 2)

Generated by the logging middleware for each request that uses cache:

- **request_id**: Unique request identifier
- **cache_hits**: Number of cache hits in this request
- **cache_misses**: Number of cache misses
- **cache_hit_rate**: Hit rate for this request
- **cache_duration_ms**: Total cache operation duration

**Example:**
```json
{
  "request_id": "abc123-def456",
  "cache_hits": 5,
  "cache_misses": 1,
  "cache_hit_rate": 0.833,
  "cache_duration_ms": 25.5
}
```

## Key Metrics

### Hit Rate
- **What**: Percentage of cache requests that returned cached data
- **Good**: > 80% (configurable via `CACHE_PERFORMANCE_ALERT_THRESHOLDS`)
- **Bad**: < 80% indicates cache is not being utilized effectively

### Penetration Rate
- **What**: Percentage of requests for non-existent resources (cache null values)
- **Good**: < 10% (configurable)
- **Bad**: > 10% may indicate malicious activity or incorrect cache keys

### Average Duration
- **What**: Average time for cache operations
- **Good**: < 10ms for in-memory cache
- **Bad**: > 100ms indicates slow Redis connection or large payloads

### Slow Operation Rate
- **What**: Percentage of operations exceeding the slow threshold (> 100ms)
- **Good**: < 5%
- **Bad**: > 20% triggers alert

## Viewing Logs

### Real-time monitoring:
```bash
tail -f backend/logs/cache.log
```

### Filter specific events:
```bash
grep "cache_performance_alert" backend/logs/cache.log
grep "cache_performance_summary" backend/logs/cache.log
```

### Parse with jq:
```bash
grep "cache_performance_summary" backend/logs/cache.log | jq '.global.hit_rate'
```

### View slow operations:
```bash
grep "Slow cache operation" backend/logs/cache.log
```

## Configuration

Alert thresholds are configured in `backend/core/settings.py`:

```python
CACHE_PERFORMANCE_ALERT_THRESHOLDS = {
    'low_hit_rate': 0.8,              # Alert if hit rate < 80%
    'high_penetration_rate': 0.1,     # Alert if penetration rate > 10%
    'slow_operation_ms': 100,         # Slow operation threshold in ms
    'high_error_rate': 0.05,          # Alert if error rate > 5%
}
```

## Troubleshooting

### Low Hit Rate
- **Symptoms**: Hit rate consistently below 80%
- **Causes**:
  - TTL too short - cache expires before reuse
  - High cache churn - data changes frequently
  - Poor cache key design - too many unique keys
- **Solutions**:
  - Increase TTL for stable data
  - Implement cache warming for hot data
  - Review cache key generation logic

### High Penetration Rate
- **Symptoms**: High percentage of null value cache hits
- **Causes**:
  - Clients requesting non-existent resources
  - Malicious scanning/attack
  - Incorrect data access patterns
- **Solutions**:
  - Implement rate limiting
  - Add request validation
  - Monitor for suspicious IP addresses

### Slow Operations
- **Symptoms**: Cache operations taking > 100ms
- **Causes**:
  - Slow Redis connection
  - Large serialized payloads
  - Network latency
- **Solutions**:
  - Check Redis server performance
  - Optimize serializer (use msgpack instead of JSON)
  - Add connection pooling
  - Consider data partitioning

## Performance Alerts

Alerts are automatically generated when thresholds are exceeded and logged to `cache.log`. The system includes **alert suppression** to prevent alert fatigue - the same alert type for the same endpoint will only be logged once every 5 minutes.

To disable specific alerts, adjust the thresholds in `settings.py`:

```python
# Disable low hit rate alerts by setting threshold to 0
CACHE_PERFORMANCE_ALERT_THRESHOLDS = {
    'low_hit_rate': 0.0,  # Never alert
    ...
}
```

## Integration with Monitoring Tools

The structured JSON format makes it easy to integrate with monitoring and alerting platforms:

### Elasticsearch + Kibana
```bash
# Ship logs to Elasticsearch
filebeat -e -c filebeat.cache.yml
```

### Prometheus + Grafana
- Individual cache operations are already exported as Prometheus metrics
- Use Grafana to visualize hit rate, duration, and penetration rate over time

### Custom Scripts
```python
import json

with open('backend/logs/cache.log', 'r') as f:
    for line in f:
        log = json.loads(line)
        if log.get('event') == 'cache_performance_summary':
            print(f"Hit rate: {log['global']['hit_rate']:.2%}")
```

## Best Practices

1. **Monitor regularly**: Review cache performance logs daily during development
2. **Set appropriate thresholds**: Adjust alert thresholds based on your application's needs
3. **Investigate alerts**: Don't ignore alerts - they often indicate real problems
4. **Track trends**: Look for gradual degradation over time, not just instant alerts
5. **Correlate with metrics**: Use Prometheus metrics alongside logs for full visibility
6. **Test under load**: Run performance tests to verify cache behavior under high traffic
