## ADDED Requirements

### Requirement: Unified cache key generation
The system SHALL use a single function `get_standard_cache_key()` to generate all cache keys across ViewSet layer, business service layer, and cache invalidation layer. This ensures consistency and eliminates the risk of cache invalidation failures due to key format mismatches.

#### Scenario: Generate global cache key
- **WHEN** a ViewSet with global data (no user-specific filtering) generates a cache key
- **THEN** the key format SHALL be `{prefix}:{view_name}[:parent_keys][:pk]`
- **AND** the key SHALL NOT contain `user_id` parameter
- **AND** the key SHALL NOT contain `SEPARATED` marker
- **EXAMPLE**: `courses:CourseViewSet:1` for course detail, `courses:CourseViewSet` for course list

#### Scenario: Generate user-specific cache key
- **WHEN** a ViewSet with user-specific data (filters by user or enrollment) generates a cache key
- **THEN** the key format SHALL be `{prefix}:{view_name}[:parent_keys][:pk]:user_id={user_id}`
- **AND** the key SHALL contain `user_id` parameter from authenticated request
- **EXAMPLE**: `courses:EnrollmentViewSet:user_id=123`

#### Scenario: Generate separated cache key (global data)
- **WHEN** a ViewSet using separated caching generates a key for global data
- **THEN** the key format SHALL be `{prefix}:{view_name}:SEPARATED:GLOBAL[:parent_keys][:pk]`
- **AND** the key SHALL contain `SEPARATED:GLOBAL` markers
- **EXAMPLE**: `courses:ChapterViewSet:SEPARATED:GLOBAL:course_pk=1`

#### Scenario: Generate separated cache key (user status)
- **WHEN** a ViewSet using separated caching generates a key for user status data
- **THEN** the key format SHALL be `{prefix}:{view_name}:SEPARATED:STATUS[:parent_keys][:pk]:user_id={user_id}`
- **AND** the key SHALL contain `SEPARATED:STATUS` markers
- **EXAMPLE**: `courses:ChapterViewSet:SEPARATED:STATUS:course_pk=1:user_id=123`

### Requirement: Automatic user_id injection for user-specific ViewSets
The system SHALL automatically detect user-specific ViewSets and inject `user_id` parameter into cache keys without requiring explicit configuration.

#### Scenario: Detect user-specific ViewSet through queryset filter
- **WHEN** a ViewSet's `get_queryset()` method filters by `user=...` or `enrollment__user=...`
- **THEN** the cache key SHALL automatically include `user_id={user_id}` parameter
- **EXAMPLE**: EnrollmentViewSet filters `queryset.filter(user=self.request.user)`, key includes `user_id=123`

#### Scenario: Detect user-specific ViewSet through permissions
- **WHEN** a ViewSet's `get_queryset()` method filters by authenticated user implicitly
- **THEN** the cache key SHALL automatically include `user_id={user_id}` parameter
- **EXAMPLE**: ProgressViewSet filters `queryset.filter(enrollment__user=self.request.user)`, key includes `user_id=123`

#### Scenario: Global data ViewSet does not include user_id
- **WHEN** a ViewSet's `get_queryset()` method does not filter by user
- **THEN** the cache key SHALL NOT include `user_id` parameter
- **EXAMPLE**: CourseViewSet has no user filter, key is `courses:CourseViewSet:1`

### Requirement: Unified cache invalidation using CacheInvalidator
The system SHALL use `CacheInvalidator` class for all cache invalidation operations in signals.py, which internally uses `get_standard_cache_key()` to ensure consistency.

#### Scenario: Invalidate single ViewSet instance cache
- **WHEN** a model instance is updated or deleted
- **THEN** the system SHALL call `CacheInvalidator.invalidate_viewset()` with prefix, view_name, and pk
- **AND** the invalidated key SHALL match the format generated by `get_standard_cache_key()`
- **EXAMPLE**: `CacheInvalidator.invalidate_viewset(prefix="courses", view_name="ExamViewSet", pk=1)` invalidates `courses:ExamViewSet:1`

#### Scenario: Invalidate ViewSet list cache
- **WHEN** a model instance is created or list-level data changes
- **THEN** the system SHALL call `CacheInvalidator.invalidate_viewset_list()` with prefix and view_name
- **AND** the invalidation pattern SHALL match the format generated by `get_standard_cache_key()`
- **EXAMPLE**: `CacheInvalidator.invalidate_viewset_list(prefix="courses", view_name="EnrollmentViewSet")` invalidates all `courses:EnrollmentViewSet:*`

#### Scenario: Invalidate separated cache global data
- **WHEN** global data (e.g., chapter content) is updated
- **THEN** the system SHALL call `CacheInvalidator.invalidate_separated_cache_global()` with appropriate parameters
- **AND** the invalidated key SHALL match the format `{prefix}:{view_name}:SEPARATED:GLOBAL:...`
- **EXAMPLE**: `CacheInvalidator.invalidate_separated_cache_global(prefix="courses", view_name="ChapterViewSet", parent_pks={"course_pk": 1})`

#### Scenario: Invalidate separated cache user status
- **WHEN** user-specific data (e.g., chapter progress) is updated
- **THEN** the system SHALL call `CacheInvalidator.invalidate_separated_cache_user_status()` with appropriate parameters
- **AND** the invalidated key SHALL match the format `{prefix}:{view_name}:SEPARATED:STATUS:...:user_id={user_id}`
- **EXAMPLE**: `CacheInvalidator.invalidate_separated_cache_user_status(prefix="courses", view_name="ChapterViewSet", user_id=123, parent_pks={"course_pk": 1})`

### Requirement: Backward compatibility with dual-key write during migration
The system SHALL maintain backward compatibility during migration by writing both old and new cache keys for a transition period of 1 week.

#### Scenario: Dual-write old and new keys for migrated ViewSets
- **WHEN** a ViewSet is migrated to use `get_standard_cache_key()`
- **THEN** the system SHALL write both the new key (via `get_standard_cache_key()`) and the old key (via `get_cache_key()`)
- **AND** both keys SHALL have the same TTL and data
- **EXAMPLE**: CourseViewSet writes `courses:CourseViewSet:1` (new) and `api:CourseViewSet:1` (old)

#### Scenario: Read from new key, fall back to old key
- **WHEN** cache is read during migration period
- **THEN** the system SHALL attempt to read from the new key first
- **AND** if the new key is a cache miss, the system SHALL fall back to read from the old key
- **AND** if the old key is a cache hit, the system SHALL write the data to the new key (async repopulation)

### Requirement: Deprecation of legacy cache key generation
The system SHALL remove the legacy `get_cache_key()` function and old Mixin classes after the 1-week migration period.

#### Scenario: Remove get_cache_key() function
- **WHEN** the 1-week migration period ends
- **THEN** the system SHALL delete `get_cache_key()` function from `common/utils/cache.py`
- **AND** the system SHALL update all references to use `get_standard_cache_key()`

#### Scenario: Remove legacy Mixin classes
- **WHEN** all ViewSets are migrated to new Mixin classes
- **THEN** the system SHALL delete `CacheListMixin` and `CacheRetrieveMixin` from `common/mixins/cache_mixin.py`
- **AND** the system SHALL keep `InvalidateCacheMixin` (still in use)

#### Scenario: Remove dual-write logic
- **WHEN** the 1-week migration period ends
- **THEN** the system SHALL remove all dual-write code from migrated ViewSets
- **AND** the system SHALL remove all `get_cache_key()` calls from dual-write logic

### Requirement: Cache key format validation
The system SHALL validate that all generated cache keys follow the standardized format to prevent inconsistencies.

#### Scenario: Validate new cache key format
- **WHEN** a new cache key is generated
- **THEN** the key SHALL follow the pattern `{prefix}:{view_name}[:markers][:parent_keys][:pk][:params][:user_id]`
- **AND** the key SHALL use colons (`:`) as delimiters
- **AND** the key SHALL NOT contain spaces or special characters

#### Scenario: Detect legacy cache key format
- **WHEN** a legacy cache key is detected (e.g., starting with `api:` prefix)
- **THEN** the system SHALL log a warning during migration period
- **AND** the system SHALL migrate the data to the new key format

### Requirement: Monitoring and observability
The system SHALL provide monitoring and logging for cache key generation and invalidation to detect issues early.

#### Scenario: Log cache key generation
- **WHEN** a cache key is generated
- **THEN** the system SHALL log the key format at DEBUG level
- **EXAMPLE**: `Generated cache key: courses:CourseViewSet:1`

#### Scenario: Log cache invalidation
- **WHEN** a cache key is invalidated
- **THEN** the system SHALL log the invalidated key at DEBUG level
- **EXAMPLE**: `Invalidated cache key: courses:ExamViewSet:1`

#### Scenario: Alert on cache hit rate degradation
- **WHEN** cache hit rate drops below 50%
- **THEN** the system SHALL trigger an alert
- **AND** the alert SHALL include the affected ViewSet and key prefix
