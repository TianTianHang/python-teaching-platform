# 增强缓存日志记录 - 阶段 3：技术设计

## Context

阶段 1 提供了基础缓存日志，阶段 2 提供了实时统计和告警。但这些数据都是瞬时或短期的，无法支持长期趋势分析和容量规划。当前系统缺少：
- 缓存性能的历史趋势（日、周、月维度）
- 热点和冷点 key 的识别
- 缓存预热的量化效果评估
- Redis 内存使用情况的深度分析

**前置条件**：
- 阶段 1 已完成，`cache.log` 中有完整的操作日志
- 阶段 2 已完成，有周期性的性能汇总日志
- Redis 正常运行，可访问 INFO 命令

**约束条件**：
- 报告生成不能影响业务性能（必须在低峰期执行）
- 日志扫描任务必须高效，不能阻塞系统
- 报告内容必须简洁明了，避免信息过载

## Goals / Non-Goals

**Goals:**
- 每日自动生成缓存效率报告，包含关键指标和趋势
- 识别 Top 100 热点缓存键和 Top 100 冷点缓存键
- 分析 Redis 内存使用情况，提供容量规划建议
- 评估缓存预热策略的效果
- 提供历史性能趋势（最近 7 天、30 天）

**Non-Goals:**
- 不实现可视化图表（可通过外部工具生成）
- 不实现自动化的容量扩容（只提供建议）
- 不实现实时的热点 key 监控（已有 Prometheus）
- 不修改缓存的业务逻辑（如自动调整 TTL）

## Decisions

### 1. 报告执行时间：凌晨 2:00（可配置）

**选择**：使用 Celery Beat 在每天凌晨 2:00 执行报告生成任务

**理由**：
- 凌晨时段业务量最低，减少对系统的影响
- 有前一天完整的数据，报告更准确
- 可在上班前完成报告，便于白天查看

**替代方案**：在凌晨 0:00 执行
- **缺点**：可能有跨日的数据遗漏

### 2. 日志扫描方式：逐行扫描并聚合

**选择**：使用 Python 逐行读取日志文件，使用字典聚合统计

```python
key_stats = defaultdict(lambda: {'hits': 0, 'access_count': 0})
with open('logs/cache.log', 'r') as f:
    for line in f:
        log_entry = json.loads(line)
        if log_entry.get('cache_key'):
            key_stats[log_entry['cache_key']]['access_count'] += 1
```

**理由**：
- 流式读取，内存占用小（约 10-50MB，与文件大小无关）
- 可处理大文件（如 1GB 的日志）
- 无需外部依赖（如 ELK）

**替代方案**：使用 ELK/Loki 查询
- **优点**：查询速度快，支持复杂聚合
- **缺点**：需要额外的组件部署
- **决策**：当前方案更轻量，未来可迁移到 ELK

### 3. 热点/冷点识别：基于访问频率排序

**选择**：统计每个 cache_key 的访问次数，按频率排序

**热点定义**：
- Top 100 访问次数最多的 key
- 访问次数 > 平均值 + 2 * 标准差

**冷点定义**：
- 访问次数 < 10 次/天
- 且缓存存在时间 > 7 天

**理由**：
- 简单直观，易于理解
- Top 100 足够覆盖主要热点
- 结合统计方法（均值+标准差）更科学

**替代方案**：使用 LRU 算法
- **缺点**：需要 Redis 配合，且只能识别最热 key

### 4. 报告格式：Markdown + 结构化日志

**选择**：同时生成两种格式的报告

1. **Markdown 格式**：便于人类阅读，可发送邮件
   ```markdown
   # 缓存日报 - 2026-03-01

   ## 关键指标
   - 全局命中率: 85.3%
   - 总请求数: 1,234,567
   - 平均延迟: 2.3ms

   ## 热点 Key
   1. api:CourseViewSet:page=1 - 56,789 次访问
   2. api:ChapterViewSet:course_id=1 - 34,567 次访问
   ```

2. **JSON 格式**：便于程序解析，存入数据库
   ```json
   {
     "date": "2026-03-01",
     "global_metrics": {...},
     "hot_keys": [...],
     "cold_keys": [...]
   }
   ```

**理由**：
- Markdown 适合快速浏览和邮件发送
- JSON 适合长期存储和后续分析

### 5. Redis 容量分析：使用 INFO 命令

**选择**：使用 Redis INFO 命令获取内存使用情况

```python
redis_client = get_redis_connection()
info = redis_client.info('memory')
used_memory = info['used_memory']
max_memory = info['maxmemory']
```

**分析内容**：
- 当前内存使用量
- 最大内存限制
- 内存使用率
- key 总数
- 平均每个 key 的大小
- 预计可容纳的 key 数量

**扩容建议规则**：
- 使用率 > 80%：建议扩容
- 使用率 > 90%：紧急扩容
- 使用率 < 30%：可考虑缩减

### 6. 趋势分析：聚合多日报告数据

**选择**：将每日报告数据持久化，用于生成趋势

**存储方案**：
```python
# 使用 Django Model 存储
class CacheDailyReport(models.Model):
    date = models.DateField(unique=True)
    hit_rate = models.FloatField()
    total_requests = models.IntegerField()
    avg_duration_ms = models.FloatField()
    hot_keys = models.JSONField()
    # ...
```

**趋势查询**：
- 最近 7 天命中率趋势
- 最近 30 天请求量趋势
- 环比/同比变化

**理由**：
- 数据库查询比日志扫描快
- 支持复杂的多维度分析
- 可保留长期历史数据

**替代方案**：直接扫描日志
- **缺点**：每次都扫描 30 天日志，耗时太长（10-30 分钟）

### 7. 管理命令：支持手动触发报告

**选择**：提供 Django 管理命令，支持手动生成报告

```bash
# 生成今日报告
python manage.py generate_cache_report

# 生成指定日期范围报告
python manage.py generate_cache_report --start 2026-02-01 --end 2026-02-28

# 只生成热点分析
python manage.py generate_cache_report --analytics-only
```

**理由**：
- 方便测试和调试
- 支持补生成历史数据
- 灵活控制报告范围

## Risks / Trade-offs

### Risk 1: 日志扫描耗时过长
**影响**：如果日志文件很大（> 1GB），扫描可能超过 5 分钟

**缓解措施**：
- 只扫描前一天 00:00:00 到 23:59:59 的日志
- 使用时间范围过滤，避免扫描全部历史
- 设置超时，超时后中断并记录告警
- 考虑使用日志轮转后的文件名（如 cache.log.2026-03-01）

### Risk 2: 内存溢出
**影响**：如果统计的 key 数量过多（百万级），可能占用大量内存

**缓解措施**：
- 只统计 Top 1000，其他的只计数不保留详情
- 使用流式处理，不在内存中保留所有 key
- 监控任务内存使用，超阈值时主动终止

### Risk 3: 报告生成失败
**影响**：报告任务失败时，缺少当天的数据

**缓解措施**：
- 任务失败时发送告警（邮件/钉钉）
- 保留原始日志，支持手动补生成
- 记录失败日志，便于排查问题
- 提供重试机制（最多重试 3 次）

### Trade-off 1: 报告详细程度 vs 性能
**选择**：报告包含 Top 100 热点和冷点，而非全部

**权衡**：
- 优点：扫描速度快，性能可控
- 缺点：可能遗漏一些重要的长尾 key
- 判断：Top 100 覆盖 80% 的访问量，足够了

### Trade-off 2: 历史数据保留时间
**选择**：数据库保留 1 年的每日报告数据

**权衡**：
- 优点：足够分析年度趋势
- 缺点：占用数据库存储（约 365 条记录，可忽略）
- 备选：如需保留更久，归档到对象存储

## Migration Plan

### 部署步骤
1. 创建 `common/analytics/` 模块，实现报告生成器
2. 实现 `CacheKeyAnalyzer` 类，用于热点/冷点分析
3. 实现 `CacheCapacityAnalyzer` 类，用于容量分析
4. 实现 `CacheTrendAnalyzer` 类，用于趋势分析
5. 创建 `CacheDailyReport` Django Model，用于存储报告数据
6. 创建 Celery 周期性任务 `generate_daily_cache_report`，每天 02:00 执行
7. 创建 Django 管理命令 `generate_cache_report`
8. 在 `core/settings.py` 中添加报告相关配置（报告时间、保留天数等）
9. 部署后验证：
   - 手动运行 `python manage.py generate_cache_report`，检查报告生成
   - 等待第二天凌晨，验证自动任务执行
   - 检查报告日志文件和数据库记录

### 回滚策略
- 报告任务是独立的，失败不影响业务
- 如性能问题，可禁用 Celery 任务
- 如报告内容有误，可删除数据库记录，重新生成

## Open Questions

1. **是否需要发送报告邮件？**
   - 当前方案只生成日志文件
   - 可选：集成邮件发送，每天早上发送给运维团队

2. **是否需要缓存预热效果的量化指标？**
   - 如计算"预热命中率"：预热后的命中率 vs 未预热的命中率
   - 需要在缓存 key 中标记是否预热

3. **是否需要支持不同时间粒度的报告？**
   - 当前只支持日报
   - 可选：增加周报、月报

4. **报告数据是否需要备份到对象存储？**
   - 当前存储在数据库
   - 可选：备份到 S3/OSS，便于长期保留
