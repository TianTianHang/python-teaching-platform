# 增强缓存日志记录 - 阶段 3：周期性报告与深度分析

## Why

阶段 1 和阶段 2 建立了基础日志记录和实时告警能力，但仍缺少长期的趋势分析和深度洞察。运维人员需要了解：
- 缓存性能随时间的变化趋势（一天、一周、一个月）
- 哪些缓存键是热点（频繁访问），哪些是冷点（很少访问）
- 缓存预热策略的效果评估
- 缓存容量规划和扩容建议
- 历史性能问题的根因分析

本阶段旨在通过周期性报告和深度分析，提供这些高级洞察能力。

## What Changes

- **实现周期性缓存报告**：每日生成缓存效率报告，包含命中率趋势、热点分析、容量使用情况
- **热点 Key 分析**：统计最频繁访问的缓存键，识别热点数据
- **冷点 Key 分析**：识别很少访问的缓存键，考虑减少缓存或调整 TTL
- **缓存预热效果评估**：分析缓存预热的命中率和效果
- **缓存容量分析**：评估 Redis 内存使用情况，提供扩容建议
- **历史性能趋势**：聚合多日数据，生成趋势图表（通过日志分析）

## Capabilities

### New Capabilities
- `cache-daily-report`: 缓存日报能力，每日生成缓存效率和使用报告
- `cache-key-analytics`: 缓存键分析能力，提供热点和冷点 key 分析
- `cache-capacity-planning`: 缓存容量规划能力，评估内存使用和扩容需求
- `cache-trend-analysis`: 缓存趋势分析能力，聚合历史数据生成趋势报告

### Modified Capabilities
- 无需修改现有 capabilities，本阶段在阶段 2 基础上增强，提供分析型能力

## Dependencies

- **依赖阶段 1**：需要基础缓存日志作为数据源
- **依赖阶段 2**：需要性能统计数据进行汇总分析
- **依赖 Redis INFO 命令**：需要从 Redis 获取内存使用、key 数量等信息

## Impact

- **代码变更**：
  - `backend/common/analytics/`：新增分析模块，包含报告生成器、热点分析器
  - `backend/common/cache_warming/tasks.py`：新增周期性报告任务
  - `backend/management/commands/`：新增管理命令，支持手动生成报告

- **新增文件**：
  - `backend/logs/cache_daily_report.log`：每日报告日志
  - `backend/logs/cache_key_analytics.log`：热点/冷点分析日志

- **定时任务**：
  - Celery Beat 任务：每天凌晨 2:00 生成日报
  - 可选：每周一上午生成周报

- **性能影响**：
  - 报告生成任务在凌晨执行，避开业务高峰
  - 日志分析可能扫描大量历史日志，耗时 1-5 分钟
  - Redis INFO 命令耗时 < 10ms，可忽略

- **存储影响**：
  - 报告日志每天约 10-50 KB，一年约 18 MB（可忽略）
  - 可选：将报告存入数据库或对象存储，便于长期保留
